# RP-Bench Methodology

How we evaluate LLMs on roleplay quality. This document is the canonical reference for every formula, sample size, judge prompt, and statistical method used in the published findings.

For the higher-level *philosophy* of the benchmark (why "measure failure" rather than "measure quality", how the failure mode taxonomy was derived), see [`EXPERIMENT_DESIGN.md`](EXPERIMENT_DESIGN.md).

---

## 1. Sources of Truth

The benchmark uses **four data sources**, each with different reliability and information content:

| Source | Type | Volume | What it provides | Limitations |
|---|---|---|---|---|
| Real RP chat sessions | Donated by users | 12 chats / ~3,700 messages | Swipe pairs (~1,600), OOC corrections (87), scene summaries | Private — only derived signals published |
| Synthetic seeds | Hand-authored | 8 standard + 20 adversarial | Reproducible test scenarios | Authors' choice of failure modes; small N per seed |
| Multi-turn sessions | Generated by harness | 336 sessions × 12 turns | LLM-judged Likert scores per session + dimension | Sonnet 4 single-judge; n=1 per (model, seed) cell |
| Community arena | Blind A/B votes | 2,013 single-turn / 223 multi-turn | Human pairwise preferences | Self-selecting voters; small per-pair counts |

**Privacy:** Raw chat content is never published. The HuggingFace dataset contains only derived signals (model_a/b strings, vote outcomes) and synthetic seeds. Voter IDs are opaque UUIDs generated server-side; no IPs, emails, or names are stored.

## 2. Single-Turn Benchmark — Four Scoring Modes

Each test model generates one response per scenario; that response is scored four different ways:

### 2.1 Standard Likert (1–5 per dimension)

For each of 27 rubric dimensions across 3 tiers (Fundamentals / Quality Control / Genre Craft), the judge produces a 1–5 score with rationale:

```
score_dim ∈ {1, 2, 3, 4, 5}
overall = weighted_mean(tier1_dims × 0.4, tier2_dims × 0.35, tier3_dims × 0.25)
```

**Limitations:**
- LLM judges have generosity bias — scores cluster at 4.0–4.5 regardless of quality
- Single-judge variance not captured (we use Sonnet 4 only for the multi-turn pass)
- Empirically validated: agrees with users only 38.7% of the time (n=75 swipe pairs)

### 2.2 Flaw Hunter (100 − deductions)

Start at 100, deduct points for each quoted flaw the judge finds. Each deduction must include exact text evidence.

```
final_score = 100
            − 15 × (n_fatal_flaws)        # agency violation, character break, wrong POV/tense
            − 8  × (n_major_flaws)        # purple prose, recycled descriptions, narrating emotions, ...
            − 3  × (n_minor_flaws)        # generic detail, samey rhythm, predictable beats, ...
            + 5  × min(n_bonuses, 3)      # specific detail, real subtext, earned surprise (capped at +15)
```

Empirical floor (across 270 multi-turn sessions): −177 (one Grok 4.1 session). Empirical ceiling: 76. Mean: 36.1, median: 42.0.

**Why it's stricter than Likert**: Likert lets the judge stay vague. The flaw hunter requires evidence per deduction, exposing the judge's actual concerns.

### 2.3 Comparative (pairwise)

Two responses to the same prompt, blind A/B, judge picks one. Used for ELO computation.

**Position bias warning:** When we ran 168 pairs with the standard Sonnet-4 comparative judge, the model in position A won **84% of the time** in pass 1, **80%** in pass 2 (with A/B swapped). Across both passes, **64% of pairs flipped their answer when ordering changed**. Single-pass pairwise judging is therefore majority noise; bidirectional evaluation (run both A→B and B→A, count a pair as decided only when the same model wins both) is mandatory.

### 2.4 Objective + Slop (rule-based)

Pure prose statistics on the response text — no judge involved:

| Metric | Formula | What it measures |
|---|---|---|
| Word count | `len(tokenize(text))` | Length |
| Unique-word ratio | `unique_tokens / total_tokens` | Vocabulary diversity (Heaps' law length-biased) |
| Bigram repetition | `1 − (unique_bigrams / total_bigrams)` | Phrase recycling |
| Sentence-length variance | `var([len(tokenize(s)) for s in sentences])` | Prose rhythm |
| AI cliché density | Curated list of ~120 phrases ("ministrations", "breath hitched", "clicked into place") + regex match per 1k tokens | Slop detection |

**Length-bias caveat:** Unique-word ratio drops with longer responses by Heaps' law (`unique ∝ N^β`, β≈0.5–0.7 for English prose). Compare models within similar length tiers, not across them.

## 3. Multi-Turn Benchmark

### 3.1 Session generation

Each session is a 12-turn (24-message) back-and-forth between:
- **Test model** playing the character (`character_name`)
- **User simulator** playing the user (`google/gemini-2.5-flash` by default)
- **Optional scripted challenge turns** at specific positions to bait failure modes

Full conversation history is passed every turn — no summarization, no sliding window. Session token budget: ~5K tokens median, ~11K max. **No context compression is applied**, so failures attributable to compression are out of scope (see deferred experiment in `EXPERIMENT_DESIGN.md`).

### 3.2 Session-level Likert judging

After all 12 turns, the full transcript is sent to the judge (Sonnet 4) with this rubric:

| Dimension | Code | Description |
|---|---|---|
| Consistency over time | S.1 | Same character traits across all turns |
| Degradation resistance | S.2 | Quality holds at turn 12 vs turn 1 |
| Narrative momentum | S.3 | Forward motion without user pushing |
| Adaptive responsiveness | S.4 | Reacts appropriately to user shifts |
| Agency respect (session) | S.5 | Never writes user's actions |
| Temporal reasoning | S.6 | Time/scene logic holds |

Plus 5 standard dimensions (anti-purple, anti-repetition, show-don't-tell, subtext, pacing).

Output: `overall ∈ [1, 5]`, plus per-dimension scores with rationales.

### 3.3 Adversarial seed pool

20 seeds in `hf_dataset/_source/adversarial_seeds.json`, organized by failure target:

| Failure target | F-code | Seed count |
|---|---|---|
| Agency violation | F1 | 4 |
| POV/tense violation | F2 | 3 |
| Lore contradiction | F3 | 1 |
| Narrative stagnation | F8 | 1 |
| Instruction drift | F12 | 3 |
| Big-card detail loss | F13 | 3 |
| Other (sycophancy / temporal / subtext / character / genre) | — | 5 |

Each seed has 2-3 scripted challenge turns at positions where the failure mode is most likely to surface (e.g. emotional climax for F1, rapid POV shifts for F2).

### 3.4 Per-failure-mode aggregation

Per (model, mode):

```
mode_mean(model, mode) = mean(overall_score for sessions on seeds in mode)
mode_floor(model, mode) = min(overall_score for sessions on seeds in mode)
mode_rank(model, mode) = ordinal rank of mode_mean among all models tested on that mode
```

Floor scores below 3.5 are flagged as catastrophic — surfaced in profile cards.

### 3.5 Per-turn binary failure detection (F1 + F2 only)

For each character turn in F1- or F2-targeted sessions, the judge answers a yes/no question with strict JSON output. Aggregated as a Wilson 95% CI on a binomial proportion:

```
failure_rate = k / n
CI_low  = ((k + 1.92) − 1.96 × √(k(1-k/n) + 0.96)) / (n + 3.84)
CI_high = ((k + 1.92) + 1.96 × √(k(1-k/n) + 0.96)) / (n + 3.84)
```

n typically 33–44 per (model, mode). F12/F13 require full system-prompt context per call and were skipped to keep this binary judging cheap.

### 3.6 Target-aware flaw hunter (per session)

Same rubric as §2.2, but the system prompt is augmented with a primer specific to the seed's failure target. E.g. for an agency_violation seed:

> *"This scene was specifically designed to test for AGENCY VIOLATIONS — situations where the model writes the user character's actions, dialogue, or internal states. Be especially vigilant for fatal-flaw 'agency_violation' deductions and quote the exact phrase that crosses the line."*

Eleven such primers (one per failure target). 270 of 336 sessions scored cleanly; 66 hit unrecoverable JSON parse errors (the judge occasionally emits invalid JSON like `"quote" (note),` despite explicit "strict JSON" instructions). Recovery patterns documented in `judge_session_flaw_hunter.py`.

## 4. Community Arena

### 4.1 Architecture

A blind A/B voting UI at `arena.l3vi4th4n.ai`. Backend is a single Next.js server with append-only JSONL vote logs in a Railway-mounted persistent volume.

Critical anti-bias features (each is a real defense against documented failure modes in earlier rounds):

| Feature | Defends against |
|---|---|
| Models hidden until vote submits | Voter brand bias |
| Random A/B side flip per voter | Position bias |
| Per-voter cookie + per-pair dedup | Single-voter ballot stuffing |
| Least-voted-pair priority queue | Coverage skew |
| Rate limit (1 vote / 3 sec, 30 / 5 min) | Click-through farming |
| 6 catch pairs at every 10th slot | Random-clicker / bot voters |
| Server-side IP-not-stored cookie | PII leakage |

### 4.2 Catch pair voter quality filter

Six hand-authored pairs where one response is unambiguously better (refusal, repetition, wrong-scene, user-hijack, truncated fragment, meta-commentary). Vote outcome is checked against the pre-declared correct answer.

```
voter_pass_rate = correct_catches / total_catches_seen_by_voter
voter_quality = "suspect"  if total ≥ 2 and voter_pass_rate < 0.5
              | "marginal" if total ≥ 2 and 0.5 ≤ voter_pass_rate < 0.75
              | "ok"       if total ≥ 2 and voter_pass_rate ≥ 0.75
```

Suspect voters' votes are excluded from leaderboard aggregation. At 2,013 votes, 2 voters were flagged suspect (one with 0/6, one with 0/2 catch pass rate).

One catch (`catch_user_hijack_cafe`) turned out ambiguous — community is split 50/50 on emotional-hijack vs literary-restraint. It's excluded from voter scoring but kept in the rotation as a standalone preference-split datapoint.

### 4.3 Frequentist ELO (100-shuffle averaged)

Per pair, decide outcome:
```
outcome(A, B) = 1.0  if A wins
              | 0.0  if B wins
              | 0.5  if tie
```

Standard ELO update with K=32:
```
E_A = 1 / (1 + 10^((R_B − R_A) / 400))
R_A' = R_A + K × (outcome − E_A)
R_B' = R_B + K × ((1 − outcome) − (1 − E_A))
```

Rerun across 100 random shuffles of the matchup order, average final ratings to reduce path-dependence:
```
ELO(model) = mean(R_model across 100 shuffles)
stability(model) = std(R_model across 100 shuffles)
```

Limitation: `stability` is shuffle variance only — it underestimates true uncertainty because it ignores the finite-sample size of the underlying votes.

### 4.4 Bayesian ELO (Bradley-Terry with Metropolis-Hastings)

The principled version. Each model has a latent skill θ_m sampled from a weakly-informative prior:

```
θ_m ∼ N(0, σ²)        with σ = 300

P(i beats j) = sigmoid((θ_i − θ_j) / scale)
            where scale = 400 / log(10)        # standard ELO scale: 400 = 10× odds

P(i ties j) = 0.5 × (P(i beats j) + P(j beats i))   # treated as half-credit each way
```

**Posterior sampling** via Metropolis-Hastings, 4 chains × (3000 burn-in + 12000 samples) = 48,000 posterior samples per model. Pure NumPy, no PyMC dependency. Acceptance rate: ~35%.

Output: ELO posterior mean + 95% credible interval per model:
```
ELO_m = 1500 + mean(θ_m posterior samples)
CI_95(ELO_m) = [percentile(1500 + θ_m samples, 2.5),
                percentile(1500 + θ_m samples, 97.5)]
```

**Width comparison:** Bayesian 95% CI is ~260 points wide at 1,857 votes; the frequentist `±stability` is only ~45 points. The frequentist version is overconfident — the Bayesian posterior reflects ALL uncertainty (prior + finite votes), not just shuffle variance.

### 4.5 SFW vs NSFW split

Scenarios are tagged. Per-model NSFW win rate = wins_on_nsfw / matches_on_nsfw. Surfaces models that handle ERP differently than SFW (e.g. DeepSeek 51% SFW vs 30% NSFW; Mistral 51% vs 67%).

## 5. Cross-Method Correlation Matrix

For every pair of methods, compute Spearman rank correlation across the 11–20 model pool:

```
r(method_a, method_b) = 1 − 6 × Σ(rank_diff²) / (n × (n² − 1))
                       where rank_diff = rank_in_a(model) − rank_in_b(model)
```

Empirical headline: **Bayesian community ELO is uncorrelated with every LLM-judge method** (rho between −0.31 and −0.07). LLM-judge methods all agree +0.6 to +0.9 with each other (they measure the same judge taste). Behavioral metrics (unique-wr, repetition) cluster together (+0.98) but only weakly correlate with anything else.

This is the single most important methodological finding: **judge-based and human-based rankings are different things**, not different views of the same thing.

## 6. Failure-Target Validation

Cross-tabulate seed `failure_target` against flaw hunter output to test whether seeds are doing what they claim:

```
P(flaw_type X | seed targets X) = sessions_on_X_seeds_with_X_flaw / total_sessions_on_X_seeds
```

Empirical: 4/11 (36%) of seed targets see the expected fatal/major flaw type in their top-3 most-frequent. The other 7/11 see `purple_prose` as the top serious flaw — meaning models default to overwriting under stress regardless of what specific failure the seed targets. Seeds work (the per-turn binary detector confirms targeted failures); the flaw hunter's `purple_prose` category is just so general it dominates aggregates.

## 7. Cost Efficiency

Quality per dollar at OpenRouter prices (60/40 input/output blended):

```
blended_cost($/1M tokens) = 0.6 × input_price + 0.4 × output_price
likert_per_dollar = likert_mean / blended_cost
fh_per_dollar = flaw_hunter_mean / blended_cost
```

Empirical: DeepSeek V4 Flash ($0.18/1M, FH 50.6) is **281× more cost-efficient** than Opus 4.7 ($39/1M, FH 42.8) on the flaw hunter metric. Frontier-tier marginal quality is dwarfed by the 200× price premium.

Caveat: Pricing as of 2026-04 from OpenRouter. Proprietary models on first-party APIs may price differently.

## 8. Statistical Conventions

### 8.1 Standard error of the mean

For a per-model metric computed from `n` independent observations:

```
SEM = √[Σ(x_i − x̄)² / (n − 1)] / √n
```

Reported alongside any `mean ± value` notation.

**Caveat (also acknowledged by vals.ai):** SEM captures sampling uncertainty in the per-cell measurement. It does **not** reflect:
- Variability across prompts / seeds / deployment settings
- Stochastic LLM generation variance (we run each session once)
- Judge variance (we use a single judge, Sonnet 4)

### 8.2 Wilson score interval (binomial proportions)

For binary failure rates:
```
W_low,high = (p̂ + z²/(2n) ± z × √(p̂(1−p̂)/n + z²/(4n²))) / (1 + z²/n)
            with z = 1.96 for 95% CI
```

Used wherever we report `4.2% [±2.1%]` for failure rates.

### 8.3 Spearman rank correlation

For non-parametric rank agreement:
```
ρ = 1 − 6 × Σ(d_i²) / (n × (n² − 1))
   where d_i = rank_X(i) − rank_Y(i)
```

Tied ranks averaged. Used in §5 (cross-method correlation) and §6 (failure-target validation).

### 8.4 What we don't have

For full transparency:
- **Multi-judge ensembles** — single judge (Sonnet 4) for everything except community arena. Inter-rater agreement (Cohen's kappa) untested.
- **Stochastic variance** — each (model, seed) cell is generated once. `temperature=0.8` means re-runs would differ; we don't capture that variance.
- **Latency** — not currently measured per model (data exists in OpenRouter logs but isn't aggregated).
- **Composite uncertainty propagation** — when we compute "avg failure rank", we treat per-mode means as point estimates and don't propagate per-mode uncertainty into the aggregate.

## 9. Bias Corrections Applied

| Bias | Where it shows up | Correction |
|---|---|---|
| Position bias | LLM-judged pairwise comparisons | Bidirectional evaluation: run A→B and B→A, count win only if same model wins both passes. Empirical bias rate: 64% on single-pass. |
| Length bias | Unique-word ratio | Acknowledged via length-tier comparisons; not corrected. |
| Click-farming | Community arena | Rate limiting (1/3 sec, 30/5 min) + per-voter dedup. |
| Bot voting | Community arena | Catch-pair voter quality filter (6 pairs, suspect threshold <50% pass rate). |
| Self-selection | Community voter pool | Acknowledged but not corrected — voters are RP-community-skewed. |
| Brand bias | Both arenas | Models hidden until vote submits. |
| Generosity / centering | Likert judging | Mitigated by also running flaw hunter (deduction-based) and community (pairwise). |
| Single-judge bias | Multi-turn judging | Acknowledged; addressing it requires multi-judge ensemble runs (~$10 for a calibration subset, deferred). |

## 10. Test-Set Privacy

**Currently:** All 20 adversarial seeds + 8 standard seeds are public (HuggingFace `lazyweasel/roleplay-bench` dataset). This means future model versions can train on them — a known problem already documented in vals.ai's methodology.

**Mitigation under consideration:** Move 4 of the 20 adversarial seeds to a private holdout, never published. Run benchmarks on both public + private sets. Compute leakage delta:

```
leakage_delta(model) = mean(score on public seeds) − mean(score on private seeds)
```

Models trained on the public seeds (post-publication retrain) would show systematically higher public scores than private. Not yet implemented — flagging here as the next methodological upgrade.

## 11. Reproducibility

Every analysis in this benchmark is reproducible from the published artifacts. Each script is idempotent and incremental (re-running skips already-completed work).

```sh
# Single-turn
rp-bench run --judge-mode standard
rp-bench run --judge-mode flaw_hunter

# Multi-turn
rp-bench multiturn --turns 12 --adversarial

# Community arena (live, hosted)
# Visit arena.l3vi4th4n.ai/arena

# Analyses
python3 analyze_community_arena.py
python3 analyze_bayesian_arena_elo.py
python3 analyze_behavioral_metrics.py
python3 judge_per_turn_failures.py
python3 judge_session_flaw_hunter.py
python3 analyze_flaw_hunter_sessions.py
python3 analyze_method_correlations.py
python3 analyze_cost_efficiency.py
python3 analyze_arena_timeseries.py

python3 analyze_model_profiles.py

```

Output files in `results/` are version-controlled snapshots; re-running with new data re-generates them. The HuggingFace dataset `lazyweasel/roleplay-bench` mirrors `hf_dataset/` (parquets + analysis JSONs).

## 13. Honest Limitations

1. **Small N per cell.** `n=1` per (model, seed) on most multi-turn cells. CIs on per-mode means are wide.
2. **Single judge for everything except community arena.** Sonnet 4's aesthetic preferences shape every Likert-derived ranking.
3. **Self-selection in voter pool.** 338 community voters are a specific RP-community demographic; findings reflect their taste, not a universal preference.
4. **English-dominant.** The benchmark has Russian seeds but the multi-turn run was English-only; Russian community arena hasn't been launched.
5. **No preset testing.** All models tested raw — no SillyTavern presets, no custom system prompts, no sampler tuning. A model that's mediocre raw may be excellent with tuning; the benchmark doesn't capture this floor-vs-ceiling spread.
6. **No long-context test.** Sessions max out at 12 turns; real RP runs 100+ turns and triggers compression. Compression-induced failures are out of scope.
7. **No latency measurement.** Speed isn't currently a benchmark axis.
8. **Test-set leakage risk.** All seeds public; future model retrains may absorb them.

These are documented in `EXPERIMENT_DESIGN.md` and the README's empirical-validation section — readers should weight rankings accordingly.
